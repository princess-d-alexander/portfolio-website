My Contributions

I implemented the reusable dialogue management system by expanding the original DialogueHandler stub that Aubrey and Ian started into a central handler for dialogue flow. I created the dialogue abstraction using DialogueList, implemented state tracking through the Seen table, added conditional logic, and integrated dialogue behavior into the game loop. I also rewrote previously repetitive conditional checks into reusable looping logic and revised the structure based on teammate pull request feedback.

Write-Up

The DialogueHandler demonstrates medium-scale abstraction because it centralizes a full gameplay subsystem into a reusable handler instead of scattering dialogue logic throughout main.lua. Before the rewrite, dialogue progression depended on repeated hardcoded conditional statements. I replaced this with a DialogueList table containing reusable dialogue entries with shared properties such as key, file, and condition. The handler then loops through these entries using pairs() to determine when dialogue should appear. This reduces duplication and makes the system easier to extend because new dialogue events can be added by inserting another table entry rather than rewriting large conditional blocks.

This abstraction also separates responsibilities across the codebase. DialogueHandler manages dialogue state, textbox progression, conditional dialogue triggers, and click behavior, while main.lua only calls the handler during update, draw, and input events. This separation improved readability and reduced overlap between gameplay logic and interface behavior. Revisions made through teammate feedback also helped ensure that the dialogue system behaved more consistently during normal gameplay and did not interfere with testing scenarios.

Write-Up

This sample demonstrates medium-scale architecture because it shows how our project separates the front-end interface into distinct components instead of placing all visual and interaction logic inside main.lua. The architecture uses separate files for rendering, asset uploading, and button information/behavior. drawingHandler.lua controls what appears on screen, ui_list.lua loads all the image assets, items_list.lua defines button positions and visibility, and meeple_button.lua handles meeple-specific interaction behavior.

This structure was necessary because our game PaperQuest is a Love2D idle game with many visual systems that change as the player progresses. Love2D does not provide a built-in interface layout system, so we had to organize our own architecture for drawing images, tracking clickable regions, and displaying changing resource values. By separating these responsibilities, I was able to redesign the UI and add the meeple system without rewriting unrelated gameplay logic. The before-and-after screenshots show how this architecture supported a clearer, more expandable interface.
