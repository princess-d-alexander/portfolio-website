Description

The component I tested was the meeple progression and production system. Meeples are worker units that can be assigned to the forest or factory, so they directly affect wood production, paper production, and game balance. My tests are scenario-based regression tests, meaning they load specific saved game states and check that important systems still behave correctly after code changes.

These tests cover numerical and progression-related behavior: resources should not become negative, assigned meeples should not be more than the total number of meeples, and production values should remain logically valid. I did not automate tests for visual layout, hitbox alignment, or rendering correctness because those depend on Love2D’s graphical output and are better checked through manual playtesting.

Testing Infrastructure

Where to Find My Work

This sample is located in the PaperGame repository on the testing-suite-princess branch. The relevant files are handlers/testingHandler.lua, items_list.lua, and the custom test save files in the tests folder.

My Contributions

I expanded the existing testing mode by adding new selectable test scenarios and reusable validation assertions. I integrated my custom save-state tests into the testing interface so they could be loaded directly from inside the game.

Code Samples

handlers/testingHandler.lua

assertValidNonnegative = function(value, label)
    assert(value ~= nil,
        label .. " should not be nil.")

    assert(type(value) == "number",
        label .. " should be a number.")

    assert(value >= 0,
        label .. " should never be negative.")
end,

Description

The project’s testing infrastructure uses an in-game testing mode controlled by testingHandler.lua. Instead of manually editing save files, we can enter testing mode and select a test button that loads a predefined gameplay state. The game then runs automated assertions during execution, which makes broken states easier to catch immediately.

My additions made this infrastructure more useful for testing meeple and production logic. The custom save files let me reproduce specific edge cases with one click, such as mid-game meeple states and resource-limited production states. This helped verify that the meeple system stayed consistent after UI and progression changes, especially because the game has many connected systems that update continuously.
