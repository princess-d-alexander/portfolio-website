Domain-Specific Architecture

Where to Find My Work

This sample is located in the PaperGame repository in handlers/dialogueHandler.lua and main.lua. It was submitted through PR #122, “Add dialogue handler with conditionals.” Relevant sections include dialogueHandler.lua lines 1–128 and main.lua lines 52–55, 80–123, and 139.

My Contributions

I designed and implemented the architecture that integrates narrative dialogue into the Love2D event-driven game loop. I built the DialogueHandler subsystem, connected it to love.update(dt), love.draw(), and love.mousepressed(), and implemented one-time dialogue triggers tied to gameplay progress such as collecting wood, making paper, purchasing weapons, unlocking meeples, and reaching prestige milestones.

Description

This code demonstrates domain-specific architecture because the structure of the system is shaped by the needs of our idle game built in Love2D. Unlike a standard application, PaperQuest continuously runs through update and draw cycles while checking player progress in real time. The dialogue system therefore needed to support narrative events that could appear dynamically when players reached milestones without interrupting the game loop.

The architecture solves the problem of coordinating gameplay progression with narrative pacing. In an idle game, values can update every frame, so dialogue prompts cannot simply check conditions repeatedly without tracking whether they have already appeared. I solved this by using the Seen table to track completed dialogue events and condition checks tied to gameplay state. The handler fits Love2D’s structure by updating text in love.update, drawing dialogue in love.draw, and handling clicks through love.mousepressed.

My Contributions

I redesigned a major portion of the game interface during the meeple integration sprint. I created custom UI assets in Canva, redesigned the visual hierarchy, repositioned resource information, grouped related mechanics spatially, and updated the rendering logic so interface elements appeared in the correct locations as gameplay systems unlocked.

Description

The redesign follows interaction design principles including visual hierarchy, grouping, consistency, and progressive disclosure. In the earlier interface, important gameplay information was spread unevenly across the screen, which made it harder to understand which systems belonged together. In the final design, the forest area corresponds to wood harvesting, the factory area corresponds to paper production, and the meeple system has its own green panels and visual indicators. This grouping makes the interface easier to scan and understand.

I considered keeping the meeple system inside the original factory and forest buttons, but that made the interface feel crowded and made the clickable regions less clear. The final design worked better because it separated resource production, meeple management, and passive information into clearer zones while preserving the game’s hand-drawn visual style. Since Love2D does not provide a built-in layout system, I also had to manually position images, text, and hitboxes so the visual design and interaction areas matched.

Engineering Quality

Where to Find My Work

This evaluation refers to the overall PaperGame codebase, especially main.lua, ui_list.lua, items_list.lua, handlers/dialogueHandler.lua, handlers/gameStateHandler.lua, handlers/perSecHandler.lua, handlers/drawingHandler.lua, and handlers/testingHandler.lua.

Visual Sample

Description

Overall, the codebase shows strong engineering quality because it is organized around clear responsibilities instead of placing all game logic in one file. The project separates assets, classes, handlers, and state files into different areas, which makes the structure easier to navigate as the game grows. main.lua acts mostly as the entry point: it loads dependencies, handles input, runs the update loop, and delegates work to handlers such as StateHandler, DrawingHandler, TestingHandler, DialogueHandler, and SoundHandler. This is strong engineering because each major system has a clear role.

The codebase also demonstrates quality through reusable patterns. Assets are centralized in ui_list.lua, game objects and buttons are centralized in items_list.lua, and production calculations are isolated in perSecHandler.lua. Many objects are stored in tables and processed with loops, which reduces repeated code. However, the codebase still falls short in places where hard-coded values and repeated conditionals remain. For example, some unlock logic could be moved into structured tables, and the save/load system would be stronger if it used labeled data instead of relying on exact line order. Given more time, I would improve the codebase by replacing fragile save files with named fields, organizing unlock rules into reusable structures, and expanding the testing system to cover more progression and UI behavior.
