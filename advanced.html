Security: Threat Model Analysis

Where to Find My Work

This analysis is based on the PaperGame repository, especially main.lua, handlers/gameStateHandler.lua, handlers/testingHandler.lua, handlers/perSecHandler.lua, and handlers/dialogueHandler.lua. These files handle user input, save/load behavior, testing validation, production calculations, and dialogue state.

My Contributions

I ran a threat model analysis on PaperQuest using the OWASP threat modeling process and Microsoft's STRIDE threat model . I focused on threats related to save-file tampering, invalid gameplay states, progression bypassing, resource calculation errors, testing-mode misuse, and repeated dialogue triggers. I also connected the analysis to code I worked on, especially validation assertions, save-state tests, production checks, and dialogue state tracking.

Surface Area

The main surface area of PaperQuest includes player mouse clicks, keyboard input, save files, testing files, UI buttons, the sound slider, and the gameplay update loop. Players interact with the system by clicking weapon buttons, factory buttons, meeple buttons, testing buttons, help buttons, and dialogue textboxes. The game also reads and writes local files such as gameSaveData.txt and testing save states, which creates a security-relevant entry point because a user could manually edit those files outside the game.

Threats and Triage

Threat Actor / Action / Attack / Asset / Objective STRIDE Category Triage
Save-file tampering A player edits gameSaveData.txt to change resource, meeple, weapon, or prestige values in order to bypass normal progression. Tampering Mitigate. The testing handler already checks for nil, non-number, and negative values. Given more time, I would add validation directly into the save/load system and reject impossible states before they enter gameplay.
Invalid meeple state A player or corrupted save file creates more assigned meeples than total meeples, damaging the consistency of production calculations. Tampering / Denial of Service Mitigate. My assertions check that forest meeples plus factory meeples do not exceed total meeples. This prevents impossible states from silently affecting the game.
Resource calculation abuse A malformed save or edge case causes paper production to exceed available wood, allowing the player to generate paper without the required resource cost. Tampering Mitigate. calculatePaper checks whether enough wood exists before producing paper. My production-limit test also checks that paper and wood remain logically consistent.
Testing-mode misuse A user enters testing mode and loads artificial save states, then exits testing mode in a way that could affect normal gameplay progression. Elevation of Privilege Accept / partially mitigate. Testing mode is a developer-facing tool, not a feature that players will interact with. The game saves before entering testing mode and reloads the saved game when exiting testing mode, which reduces the risk of test states replacing normal progress.
Repeated dialogue triggers A progression condition remains true every frame and repeatedly triggers the same dialogue, interrupting gameplay or making the interface unusable. Denial of Service Mitigate. DialogueHandler.Seen tracks which dialogue events have already appeared so each event only happens once.

Description

I used the OWASP Threat Modeling Process by first scoping the system, identifying entry points, and listing assets that need protection. I also used the Carnegie Mellon SEI threat-scenario structure of ACTOR, ACTION, ATTACK, ASSET, EFFECT, and OBJECTIVE to describe how attacks could affect PaperQuest. (SEI Threat Modeling Guide) For PaperQuest, the most important assets are valid save data, resource values, progression state, meeple assignments, unlock conditions, and stable gameplay. The largest risks come from local file editing because the game stores progress in plain text and loads values directly into gameplay variables.

The strongest threats are tampering and invalid state creation. A player could edit save data to create impossible values, bypass progression, or crash the game by loading nil or nonnumeric values. The current project partially mitigates these risks through validation assertions, production checks, and scenario-based regression tests. These checks make sure values are not nil, are numeric, are nonnegative, and follow important gameplay rules such as assigned meeples not exceeding total meeples.

The analysis also shows where the project falls short. The testing system catches invalid states during test mode, but the main save/load system still trusts the save file too much. Given more time, I would move the validation checks into StateHandler.loadGame so corrupted or edited save files are rejected before they affect gameplay. I would also replace the positional save format with labeled fields, add default fallback values for missing data, and separate developer testing tools from normal player-facing gameplay.

Description

The redesign follows interaction design principles including visual hierarchy, grouping, consistency, and progressive disclosure. In the earlier interface, important gameplay information was spread unevenly across the screen, which made it harder to understand which systems belonged together. In the final design, the forest area corresponds to wood harvesting, the factory area corresponds to paper production, and the meeple system has its own green panels and visual indicators. This grouping makes the interface easier to scan and understand.

I considered keeping the meeple system inside the original factory and forest buttons, but that made the interface feel crowded and made the clickable regions less clear. The final design worked better because it separated resource production, meeple management, and passive information into clearer zones while preserving the game’s hand-drawn visual style. Since Love2D does not provide a built-in layout system, I also had to manually position images, text, and hitboxes so the visual design and interaction areas matched.

Engineering Quality

Where to Find My Work

This evaluation refers to the overall PaperGame codebase, especially main.lua, ui_list.lua, items_list.lua, handlers/dialogueHandler.lua, handlers/gameStateHandler.lua, handlers/perSecHandler.lua, handlers/drawingHandler.lua, and handlers/testingHandler.lua.

Description

Overall, the codebase shows strong engineering quality because it is organized around clear responsibilities instead of placing all game logic in one file. The project separates assets, classes, handlers, and state files into different areas, which makes the structure easier to navigate as the game grows. main.lua acts mostly as the entry point: it loads dependencies, handles input, runs the update loop, and delegates work to handlers such as StateHandler, DrawingHandler, TestingHandler, DialogueHandler, and SoundHandler. This is strong engineering because each major system has a clear role.

The codebase also demonstrates quality through reusable patterns. Assets are centralized in ui_list.lua, game objects and buttons are centralized in items_list.lua, and production calculations are isolated in perSecHandler.lua. Many objects are stored in tables and processed with loops, which reduces repeated code. However, the codebase still falls short in places where hard-coded values and repeated conditionals remain. For example, some unlock logic could be moved into structured tables, and the save/load system would be stronger if it used labeled data instead of relying on exact line order. Given more time, I would improve the codebase by replacing fragile save files with named fields, organizing unlock rules into reusable structures, and expanding the testing system to cover more progression and UI behavior.
